// 01. Overview
What This System Does and Why

Most retail quantitative systems stop at backtesting. This one didn't. Every component from data ingestion to live order submission is written, owned, and running in production. Each stage does one thing and passes its output forward. Complexity is at the system level, not buried inside any individual module. That makes each piece independently testable and replaceable, which matters a lot when something breaks at 9:45am on a trading day.(or IBKR resets my password 5 minutes before market open)

What makes this non-trivial

The system filters ~10,000 SEC-registered securities down to ~4,200 investable equities, pulls daily price history for all of them via a custom IBKR async connection pool, engineers 50+ alpha features per ticker across all CPU cores, predicts next-day direction with a calibrated XGBoost classifier, validates every strategy idea through a backtester tracking 20+ risk metrics, runs a news scan via Claude in research mode, and submits four-legged bracket orders with automated downside protection. All of this finishes before market open each morning.

Why long-only equities, no ETFs, no REITs, no banks

The prediction target is next-day directional move for individual operating companies. ETFs introduce index replication noise. REITs are driven by cap rates and distribution schedules. Banks have opaque balance sheets where standard momentum and mean-reversion signals behave erratically. SPACs, leveraged products, and shell companies are rejected at the ticker universe stage before any price data is even pulled. The filter keeps the training set in a regime where the features are actually informative. Somewhat ironic given the context of this document, but that's what the data said.

// 03. Stage 1: Ticker Universe
Building a Clean Equity Universe from SEC Data

Every night the system pulls the SEC's company_tickers_exchange.json endpoint, the same registry regulators use. Raw output contains 10,000+ entries: operating companies, SPACs, ETFs, leveraged products, shell companies, funds, warrants. The filtering stage cuts that down to ~4,200 investable equities using two passes: keyword rejection first, then regex patterns for anything keywords miss.

Layer 1: Instant-Reject Keywords

Any company whose name contains one of these words is rejected before further processing. This catches the vast majority of non-equity garbage in one O(n) pass:

acquisitionspacetffund reitwarrantpreferredholdings vehicleventuresincomeblockchain metaversecryptomergerbitcoin
Layer 2: Regex Pattern Filters

A second pass applies regex patterns for cases keywords miss. Catches leveraged products, series/class designations, numbered SPAC variants, and known fund provider names:

vanguardisharesspdrblackrock \d+x\sleveragedinverseseries [a-z] class [a-z]shell companyblank checkroman numerals after corp/inc
Why this matters for signal quality

A directional equity predictor trained on a contaminated universe leaks ETF replication behavior, SPAC announcement dynamics, and fund distribution effects into the feature space. These have fundamentally different return drivers than operating companies. Filtering hard at the universe stage, before any price data is downloaded or any feature is computed, keeps the downstream feature matrix consistent across market regimes. The cost is a few false positives on legitimate companies with unfortunate names. That tradeoff is worth it.

// 04. Stage 2: Price Ingestion
Async IBKR Connection Pool: ~20 Tickers/Second

Pulling daily price history for 4,200 tickers via Interactive Brokers TWS is a rate-limiting problem. IBKR throttles requests per connection, but allows multiple simultaneous connections each with independent limits. The solution here is a custom async connection pool backed by Python's deque, holding a set of live IB connections that workers check out, use, and return. You won't find this pattern documented anywhere online for IBKR. It was built from scratch.

Architecture

A ConnectionPool class manages up to 8 simultaneous IBKR connections. Each request checks out a connection from the deque, performs the download, and returns it. Connections that fail health checks are silently replaced.

Concurrency

Up to 32 async workers run simultaneously via asyncio.Semaphore. Tickers are batched and dispatched with asyncio.gather(). Failed tickers are retried up to 3 times with exponential backoff.

Resilience

Progress is persisted to a JSON file after each ticker. Runs can be interrupted and resumed without re-downloading completed tickers. Stale connections trigger automatic reconnect. Error codes from TWS are suppressed selectively (contract-not-found vs genuine errors).

// 2__PriceDownloader.py | connection pool core (simplified)
class ConnectionPool:
    def __init__(self, host, port, max_connections=8):
        self.connection_queue = deque()          # double-ended queue of live IB connections
        self.lock = asyncio.Lock()               # prevents concurrent pool mutations
        self.connection_health = {}               # tracks failure/success per connection

    async def get_connection(self):
        async with self.lock:
            while self.connection_queue:
                tag, ib = self.connection_queue.popleft()
                if ib.isConnected():
                    return tag, ib           # healthy connection, check it out
            return await self._create_new_connection()   # pool exhausted, open new

// 05. Stage 3: Alpha Engineering
50+ Features Engineered Across All CPU Cores

3__AlphaSensitivity.py transforms raw OHLCV bars into a feature matrix for each ticker. It also pulls live market context directly from IBKR TWS (VIX, SPY, QQQ, IWM, DIA) to compute features that place each stock in context of the current regime. All ticker-level computation runs in a ProcessPoolExecutor that saturates every available CPU core.

DVAMR Probability: a custom feature

Dynamic Volatility Adjusted Mean Reversion Probability is a custom composite signal that produces a 0–1 probability score for mean reversion conditions. It combines:

1

VIX Regime

Current VIX vs. its own rolling z-score and VIX-of-VIX

2

Volatility Normalization

Stock's return deviation normalized by its own rolling volatility

3

Market Stress Composite

Multi-severity stress indicator (moderate / severe / extreme) weighted by volume ratio

Why visibility graphs on price series?

A Natural Visibility Graph (NVG) converts a price series into a graph: each bar is a node, and two nodes are connected if one can "see" the other without a taller bar in between. The resulting graph's topology (degree distribution, clustering coefficient, mean path length) captures structural properties that FFT and rolling statistics miss, things like trend persistence, local complexity, and dominant peak structure. These features come from the academic literature on graph-theoretic time series analysis and add a genuinely distinct information channel to the feature matrix.

// 06. Stage 4: Prediction
XGBoost Binary Classifier with Isotonic Calibration

The predictor answers one question per ticker per day: will this stock close higher tomorrow than it closes today? That's it. The simplicity is deliberate, and it came from running experiments where making the target more complex made the model measurably worse.

Simple target outperforms complex target Empirical Finding

An earlier version of this system tried to push trading rules directly into the prediction label: encoding position sizing, stop logic, and exit conditions so the model could learn better trading decisions in one shot. AUC-PR dropped consistently every time the label got richer. What was happening: a complex label increases label variance without adding signal. Two identical entry conditions can produce completely different outcomes depending on where the stop sits or how long the trade runs. That variance goes into the training labels as noise. A simple binary directional label has the lowest possible noise and the clearest relationship to the input features. Trading logic belongs in the strategy layer. The predictor just calls direction.

Why AUC-PR instead of AUC-ROC?

Stocks go up slightly less than half the time after fees, so the training data is class-imbalanced. AUC-ROC is insensitive to class imbalance and scores well even for a model that performs poorly on the minority class. AUC-PR (area under precision-recall curve) specifically measures performance on positive cases. Early stopping on AUC-PR means the model stops when it stops improving precision on actual up-moves, which is the right objective for a trading signal.

Why isotonic calibration?

Raw XGBoost scores are not probabilities. A score of 0.65 does not mean 65% chance of going up. Isotonic regression maps raw scores to empirically calibrated probabilities on a held-out set. A 2% confidence margin on top of the decision threshold cuts out marginal calls. The output is a number the backtester and execution engine can treat as an actual probability estimate rather than just a ranking signal.

// 07. Stage 5: Backtesting
Research-Grade Backtesting with 20+ Metrics

The backtester is where strategy ideas get tested and broken before any real capital is at risk. It runs on Backtrader with an exact replica of the IBKR commission structure: $0.0035/share, $0.35 minimum, 1% value cap, plus exchange fees. Every metric prints to the terminal in color, graded from red to green. One exception: values so exceptional they qualify as "Unicorn" render in cornflower blue. That label comes from Van Tharp's SQN framework, which turned out to be a useful reference point for calibrating what a genuinely good result looks like.

The System Quality Number measures mean trade return divided by its standard deviation, scaled by the square root of trade count. Every metric in the terminal maps to a continuous red-to-green gradient. SQN values above 15 get a separate cornflower blue designation called Unicorn. It comes up rarely enough that when it shows up, it actually means something.

Dynamic thresholds: context-aware colorization

Fixed red/green thresholds on every metric would be misleading. A 150%+ annualized-return strategy will have higher volatility than a 20% strategy, and that's fine. The backtester computes acceptable volatility bands, VaR bounds, and drawdown limits at runtime based on Sharpe, Sortino, and raw return. Color coding is always relative to what the specific strategy is actually doing, so a high-return system doesn't look broken just because it has numbers that would be bad in a different context.

// 08. Stage 6: LLM Macro Filter
Claude in Research Mode as a Soft Filter

Quantitative signals have a blind spot for catastrophic binary events. A company filing for bankruptcy, announcing an emergency merger, or caught in a geopolitical freeze looks completely normal in the feature matrix right up until it doesn't. Price history can't predict these. Regex can't catch them reliably. This stage uses Claude (Sonnet) in research mode to scan recent news on each signal ticker before the day's orders go out.

What the LLM checks

The model is prompted to search recent news (last 14 days for news, up to 60 days for M&A) and flag any of the following:

  • Active or announced M&A (has_active_ma)
  • Bankruptcy filing or imminent insolvency
  • Geopolitical event affecting the company directly
  • Regulatory action, trading halt, or exchange delisting
  • Material crisis event (fraud, criminal investigation, etc.)
Engineering around the LLM

The LLM is not trusted for quantitative judgement. Its job is to catch qualitative flags a human analyst would spot in two minutes of reading headlines. Several engineering decisions keep this practical:

  • Per-ticker results cached to JSON; interrupted runs resume without re-querying
  • Errored tickers (rate limit, network) tracked separately and retried on resume
  • --skip-claude flag bypasses LLM for cheapest/fastest run
  • Hardcoded quantitative pre-filters run first (min price $2.00, min volume 206k)
// 09. Stage 7: Live Execution
Four-Legged OCA Brackets on a Personal Computer

The execution engine runs on a personal computer. Network outages happen. Power goes out. Open positions need protection even when the local software is completely offline. The approach is to submit all exit orders as server-side IBKR orders grouped in an OCA (One-Cancels-All) group at the time of entry. Once transmitted, IBKR holds them. The local machine can go dark and the downside protection stays live at the exchange.

Four-legged bracket structure OCA Group

Every position entry generates four orders submitted as a single atomic bracket. All three exit legs share one OCA group name (bracket_{symbol}_{parent_id}). The moment any exit fills at the exchange, IBKR cancels the remaining two. The local Python process doesn't need to be running. This is how the system works safely on consumer hardware.

Leg 1: Parent Entry (IBKR Adaptive Algo)

A limit order using IBKR's Adaptive algorithm at Normal priority. The Adaptive algo submits inside the spread and walks toward the ask over time, so fills land at mid-price or better on liquid names. Expected slippage drops from 10-15 bps with a plain limit-at-ask down to around 2-5 bps. The limit price is a safety ceiling, not the expected fill price. Slippage at each fill is logged to CSV to build a running empirical record.

Leg 2: Take-Profit (LMT, GTC)

A limit sell at the backtester-derived target price, or at entry + 2× risk if no target is provided. Good-till-cancelled and valid outside regular hours so after-hours announcements trigger the exit. OCA type 1: fills this, cancels the trailing stop and hard stop.

Leg 3: Trailing Stop (TRAIL, GTC)

A percentage-based trailing stop computed from ATR at signal time. Floor 1.2%, ceiling 4.0%. These are intentionally tighter than the backtester (1.5%/5.0%) to give back less from peak in live trading. Stays server-side, rises with the price, never falls.

Leg 4: Hard Stop (LMT, GTC)

A fixed limit order below the trailing floor. Covers the case where the trailing stop gets gapped over during a halt or an after-hours move. Sits in the same OCA group and is tighter than the trailing floor, so it only fires when something genuinely unexpected happens.

// 10. Design Decisions
Key Engineering and Research Choices

Several decisions in this system are non-obvious. This section explains the ones that come up most often when walking someone through the codebase.

Temporal data splits with 5-day embargo gaps ML

Standard train/test splits chosen at random introduce lookahead bias in financial time series. A row from 2022 in the test set alongside a row from the same week in training means both share the same market regime, making the split artificially easy. This system uses strictly temporal splits where all training data precedes all test data, with a 5-day embargo gap between them. The embargo prevents the model from learning on data that is temporally adjacent to, and therefore correlated with, the test period. This approach comes from Advances in Financial Machine Learning (Lopez de Prado) and is standard practice in serious quantitative ML.

Why not more complex targets for XGBoost? Architecture

When the prediction target was enriched to encode exit rules, sizing logic, and multi-day holding behavior, measured AUC-PR dropped consistently across experiments. The explanation is that a complex target increases label noise without increasing signal: the variance in realized outcomes from identical entry conditions (due to stop placement, holding period, market microstructure) adds irreducible noise to the training labels. A simple binary directional target has the lowest possible label noise and the clearest relationship to the input features. Complexity belongs in the strategy layer, not the prediction layer.

Zero-slippage assumption is defensible Execution

Backtests with zero slippage get dismissed as unrealistic. For this system the assumption holds up: the IBKR Adaptive Algo fills limit orders at mid-price or better on liquid names. Empirical slippage logging from live trading confirms fills landing at 2-5 bps, well inside the noise floor of daily return variance. Commissions are modeled exactly ($0.0035/share, $0.35 minimum), and fills are treated as occurring at the signal price. That is consistent with what the execution data actually shows.

Why exclude banks, REITs, ETFs? Universe

OHLCV-derived alpha factors work well for industrial, technology, consumer, and healthcare companies. Banks are a different problem: their returns correlate more with credit spreads and rate regimes than with the price-based features this system measures, and their balance sheets are opaque enough to make standard signals unreliable. REITs are driven by distribution yield dynamics. ETFs add index arbitrage and replication mechanics to the mix. Removing all of these concentrates the model's training signal where the features are actually informative.

Recency-weighted isotonic calibration ML

Standard isotonic calibration weights all calibration samples equally. This system applies recency weighting so recent observations carry more influence. Market regimes shift. A calibration curve fit on 2019 data produces miscalibrated probabilities in 2024. Recency weighting keeps the probability outputs tracking the current relationship between raw scores and realized outcomes rather than a stale historical average.

// 11. Academic Foundations
The Research That Shaped This System

Five books contributed specific techniques that appear directly in the code. This section covers what each one actually contributed rather than listing them as general influences.

01
Advances in Active Portfolio Management
Richard C. Grinold & Ronald N. Kahn
The IC framework for scoring alpha factors, the Fundamental Law of Active Management, and the concept of breadth as independent bets all fed directly into the alpha factor engineering stage. The asset correlation work in 0__AssetCorrolator.py draws from the covariance estimation methods in this book specifically.
02
Machine Learning for Algorithmic Trading
Stefan Jansen
The feature engineering approach (rolling windows, lag features, cross-sectional normalization) follows patterns from this book. The AUC-PR evaluation methodology and the framing of prediction as a ranking problem rather than a regression problem both appear directly in this system. Practically focused and worth reading before touching any financial ML work.
03
Advances in Financial Machine Learning
Marcos Lopez de Prado
The source of the temporal split methodology with embargo gaps used in this system. De Prado's demonstration that random cross-validation is inappropriate for financial time series, and that walk-forward splits with purging and embargo are the correct approach, directly motivated the training pipeline. The meta-labeling discussion (predicting trade quality rather than just direction) also influenced the decision to keep the prediction target simple.
04
Dynamic Hedging: Managing Vanilla and Exotic Options
Nassim Nicholas Taleb
No options in this system. The book contributed anyway. Taleb's treatment of tail risk and the inadequacy of variance as a complete risk description informed the decision to track Ulcer Index, CVaR, and Risk of Ruin in the backtester alongside standard volatility measures. The priority on downside protection (bracket orders, hard stops, server-side exits) comes from that same orientation.
05
Assessing and Improving Prediction and Classification
Timothy Masters
Probably the least known book on this list and worth tracking down. Masters covers how to avoid fooling yourself with in-sample performance, how to design robust out-of-sample tests, and how calibration affects the practical usefulness of a probabilistic predictor. That material directly shaped the AUC-PR methodology, the calibration stage, and the confidence margin on the decision threshold. His approach of treating prediction quality as multi-dimensional rather than a single number also influenced the metrics system in the backtester.
